# Security Considerations

本ドキュメントでは、本リポジトリにおける**セキュリティ設計上の考慮点と判断理由**をまとめます。\
本プロジェクトは技術サンプルですが、「なぜその構成を選んだのか」を明示することで、実務への展開可能性を示すことを目的としています。

---

## 基本方針

- PaaS（Cloud Run）のセキュリティ特性を前提にする
- コンテナの攻撃面を最小化する
- Laravel 標準のセキュリティ機構を尊重する
- **“やらないこと”を明確にする**
- `.env` などの秘匿情報は **イメージに含めない**

---

## Docker Hardened Images（DHI）を採用する理由

本プロジェクトでは **Cloud Run 用の prod イメージ**に Docker Hardened Images（DHI）を使用します。

### 採用理由

- ベース OS・パッケージが最小構成
- 不要なツール・デーモンを排除
- 供給元が明確で、透明性（SBOM / provenance）を意識しやすい

技術サンプルであっても、**意図を持った選択**を行うことを重視します。

---

## ローカル開発で DHI を必須にしない理由

ローカル開発は互換性と再現性のバランスを取り、Dockerfile の `dev-local` ターゲットで **Official Image** を利用します。

- 開発環境（OS / Docker 実行基盤）によっては DHI をそのまま実行できないケースがある
- チーム開発や検証の立ち上がりで「まず動く」を優先する

ただし **Cloud Run に載せる最終成果物（prod）は DHI non-dev** を前提とし、実運用想定のセキュリティ特性はそこに集約します。

---

## non-dev イメージ運用の前提

non-dev イメージは一般に以下の制約があります。

- シェルが存在しない（または限定的）
- OS パッケージマネージャが使えない
- 実行バイナリが限定される

そのため、本プロジェクトでは以下を前提とします。

- 依存関係（Composer）は **ビルド時に解決し、成果物を COPY** する
- 起動時に “環境に依存する作業” を持ち込まない（migrate 等は責務分離を検討）
- `.env` は Cloud Run の環境変数で管理する

---

## Cloud Run のセキュリティ前提

Cloud Run は以下の特徴を持ちます。\
本プロジェクトでは、これらを前提に設計しています。

- コンテナは短命である
- インスタンスは自動スケール・再作成される
- OS への直接アクセスは不可

### Cloud Run の公開範囲（unauthenticated）に関する方針

本プロジェクトでは Cloud Run を **原則 private（unauthenticated なし）** とします。

- 公開（`allUsers` に `roles/run.invoker`）はデフォルトでは行わない
- CI/CD から公開設定は変更しない（意図せぬ公開を防ぐ）
- 公開が必要な場合のみ、運用者が手動で IAM を調整する

### 具体的な対応

- ファイル永続化を前提としない
- インスタンス固有情報を持たない
- 外部ストレージ依存を最小化

---

## SQLite を採用している理由とリスク

### 採用理由

- 技術サンプルとして構成を単純化するため
- DB サーバ管理をスコープ外とするため

### リスクと割り切り

- コンテナ再作成でデータ消失
- 同時アクセス耐性が低い

これらは **明示的に許容**しており、README および設計ドキュメント（`docs/architecture.md`）に記載しています。

---

## Laravel 側のセキュリティ前提

### 利用している標準機構

- CSRF Protection
- Route Model Binding
- Policy / Gate
- Validation

### 本サンプルで扱わないもの

- 認証基盤（SSO / OAuth 等）
- 監査ログの完全実装
- 暗号鍵ローテーション

これらは **実運用で追加されるべき要素**として除外しています。

---

## 環境変数管理

- 機密情報はリポジトリに含めない
- Cloud Run の環境変数で注入
- `.env` はローカル専用

特に注入する環境変数として以下は必須項

- `APP_KEY`
- `APP_ENV`
- `APP_DEBUG=false`

---

## ログと可観測性

- `LOG_CHANNEL=stderr`
- Cloud Run の標準ログ収集に委譲
- アプリ内での独自ローテーションは行わない

---

## セキュリティ設計のまとめ

- Cloud Run の責務分界を理解した構成
- DHI による攻撃面の縮小
- Laravel 標準機構の活用
- **「簡略化している点」を明示**

本リポジトリは**安全に作るとはどういうことかを説明できる技術サンプル**であることを目指しています。
