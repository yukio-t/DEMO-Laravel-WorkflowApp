# Security Considerations

本ドキュメントでは、本リポジトリにおける**セキュリティ設計上の考慮点と判断理由**をまとめます。\
本プロジェクトは技術サンプルですが、「なぜその構成を選んだのか」を明示することで、実務への展開可能性を示すことを目的としています。

---

## 基本方針

- PaaS（Cloud Run）のセキュリティ特性を前提にする
- コンテナの攻撃面を最小化する
- Laravel 標準のセキュリティ機構を尊重する
- **“やらないこと”を明確にする**

---

## Docker Hardened Images（DHI）を採用する理由

本プロジェクトでは Docker Hardened Images（DHI）を使用しています。

### 採用理由

- ベース OS・パッケージが最小構成
- 不要なツール・デーモンを排除
- Docker 公式提供による透明性（SBOM / provenance）

技術サンプルであっても、**「とりあえず公式イメージ」ではなく、意図を持った選択**を行うことを重視しています。

---

## `-dev` イメージを使用している理由

開発環境では `-dev` イメージを使用しています。

### 理由
- シェルが利用可能
- Laravel のビルド・キャッシュ生成が容易
- 初期構築・検証のハードルが低い

これは **利便性を優先した段階的選択**であり、本番（Cloud Run）運用では以下を検討対象とします。

- non-dev イメージへの移行
- 起動時処理のビルド時前倒し
- Cloud Run Job への責務分離（migrate 等）

---

## 本番環境（Cloud Run）への `non-dev` イメージへの移行指針

non-dev イメージは以下の制約があります。

- シェルが存在しない
- OS パッケージマネージャが使えない
- 実行バイナリが限定される

そのため、以下が前提となります。

- すべての依存関係はビルド時に確定
- 起動時に行う処理を極力なくす
- 環境変数のみで動作が完結する設計

---

## Cloud Run のセキュリティ前提

Cloud Run は以下の特徴を持ちます。\
本プロジェクトでは、これらを前提に設計しています。

- コンテナは短命である
- インスタンスは自動スケール・再作成される
- OS への直接アクセスは不可

### Cloud Run の公開範囲（unauthenticated）に関する方針

本プロジェクトでは Cloud Run を **原則 private（unauthenticated なし）** とします。

- 公開（`allUsers` に `roles/run.invoker`）はデフォルトでは行わない
- CI/CD から公開設定は変更しない（意図せぬ公開を防ぐ）
- 公開が必要な場合のみ、運用者が手動で IAM を調整する

### 具体的な対応

- ファイル永続化を前提としない
- インスタンス固有情報を持たない
- 外部ストレージ依存を最小化

---

## SQLite を採用している理由とリスク

### 採用理由

- 技術サンプルとして構成を単純化するため
- DB サーバ管理をスコープ外とするため

### リスクと割り切り

- コンテナ再作成でデータ消失
- 同時アクセス耐性が低い

これらは **明示的に許容**しており、README および設計ドキュメント（Architecture.md）に記載しています。

---

## Laravel 側のセキュリティ前提

### 利用している標準機構

- CSRF Protection
- Route Model Binding
- Policy / Gate
- Validation

### 本サンプルで扱わないもの

- 認証基盤（SSO / OAuth 等）
- 監査ログの完全実装
- 暗号鍵ローテーション

これらは **実運用で追加されるべき要素**として除外しています。

---

## 環境変数管理

- 機密情報はリポジトリに含めない
- Cloud Run の環境変数で注入
- `.env` はローカル専用

特に注入する環境変数として以下は必須項

- `APP_KEY`
- `APP_ENV`
- `APP_DEBUG=false`

---

## ログと可観測性

- `LOG_CHANNEL=stderr`
- Cloud Run の標準ログ収集に委譲
- アプリ内での独自ローテーションは行わない

---

## セキュリティ設計のまとめ

- Cloud Run の責務分界を理解した構成
- DHI による攻撃面の縮小
- Laravel 標準機構の活用
- **「簡略化している点」を明示**

本リポジトリは**安全に作るとはどういうことかを説明できる技術サンプル**であることを目指しています。
